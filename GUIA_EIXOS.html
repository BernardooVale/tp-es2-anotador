Os dois eixos no código — guia didático com exemplos

Companheiro do CODEBOOK.md e do manual de adjudicação. Todos os exemplos são inventados e mínimos — o objetivo é treinar o olho para reconhecer cada padrão nos pares reais.

Atalhos: Como ler um par · Eixo A (A1–A6) · Eixo B (R1–R5) · Colinha final
A régua em uma linha: genuina = o after é seguro (Eixo A: nenhum chamador perceberia a troca) E é uma boa aula (Eixo B: o smell foi resolvido de verdade, e só ele foi mexido). Falhou um dos dois → lixo. incerto só quando o trecho não permite decidir.

0. Como ler um par (antes de qualquer eixo)

A maior parte dos nossos erros da rodada 1 veio de ler o corpo da função primeiro e formar opinião pela "cara" do diff. Inverta a ordem:

  1. Linha def dos dois lados — parâmetros, defaults, nomes (30 segundos, pega metade dos A1).
  2. Todos os return/yield/raise dos dois lados — o que sai da função e em que condições.
  3. Nomes novos do after — cada função/constante/atributo que não existia no before: está definido no trecho?
  4. Só então leia o corpo, comparando os caminhos de execução.

Eixo A — o after é seguro? (eliminatório)

Pergunta-guia: "se eu trocasse o before pelo after em produção, algum chamador, log, teste ou usuário perceberia?" Perceberia = lixo / muda_comportamento.

A1 — Assinatura e contrato falha típica

BEFORE
def parse_config(path, strict=True):
    cfg, avisos = _ler(path, strict)
    return cfg, avisos, time.time() - t0
AFTER
def parse_config(path, strict=False):
    cfg, avisos = _ler(path, strict)
    return cfg

Duas quebras numa olhada: o default mudou (strict=True→False: todo chamador que não passa strict muda de comportamento silenciosamente) e o retorno encolheu (quem fazia cfg, avisos, dt = parse_config(...) agora explode com TypeError).

compare as duas linhas def token por token; depois liste os return de cada lado e compare quantidade e estrutura do que sai. Tipo conta: lista→generator, tupla→dict, int→Enum — tudo é contrato.

A2 — Símbolo indefinido no mínimo incerto

BEFORE
def fechar_pedido(pedido):
    total = 0
    for item in pedido.itens:
        total += item.preco * item.qtd
    return total
AFTER
def fechar_pedido(pedido):
    total = _somar_itens(pedido.itens)
    return aplicar_descontos(total, pedido.cupons)

_somar_itens e aplicar_descontos não estão definidos no trecho e não existiam no before. O segundo é pior: usa pedido.cupons, que o before nunca tocou — então nem dá para afirmar que o resultado é o mesmo. Sem ver esses corpos, o comportamento é inverificável.

faça a lista dos nomes que o after usa e o before não usava. Para cada um: a definição aparece no próprio after? Não? → o par não é autocontido. Se a chamada nova lê dados que o before ignorava (pedido.cupons), suspeite de mudança de comportamento, não só de incerteza.

A3 — Curto-circuito engolindo efeito colateral falha típica

BEFORE
mudou_origem = checar_origem(inst)      # sempre executa
mudou_prot   = checar_protecao(inst)    # sempre executa
mudou = mudou_origem or mudou_prot
AFTER
mudou = checar_origem(inst) or checar_protecao(inst)
# se checar_origem devolve True,
# checar_protecao NUNCA roda nesta instância

Em Python, f() or g() não chama g() quando f() retorna algo verdadeiro. Se as funções têm efeito colateral (gravam, corrigem, logam), o after pula trabalho que o before fazia.

ache todo or/and novo entre chamadas de função. Pergunte: essas funções só calculam, ou também fazem algo? Se fazem, é mudança de comportamento. (Se forem puras, a fusão é equivalente — não rejeite por reflexo.)

A4 — Early-return que pula o "tail" comum a mais traiçoeira

BEFORE
def calcular(pedido):
    if pedido.tipo == "expresso":
        total = _base(pedido) + 15
    else:
        total = _base(pedido)
    # tail comum aos DOIS ramos:
    total *= pedido.cambio
    registrar(total)
    return total
AFTER
def calcular(pedido):
    if pedido.tipo == "expresso":
        return _total_expresso(pedido)
    total = _base(pedido)
    total *= pedido.cambio
    registrar(total)
    return total

O ramo "expresso" agora retorna antes do tail: não multiplica pelo câmbio e não registra. A extração parece limpa — e está errada (a menos que _total_expresso repita o tail, o que A2 manda verificar). Foi exatamente o padrão do r1_000001, que o próprio upstream corrigiu depois.

quando um ramo virar return helper(...), olhe o que existia depois do if/else no before. Tinha código comum (multiplicação, log, acumulação, conversão)? Então aquele ramo perdeu esse código.

A5 — Exceções e avisos falha típica

BEFORE
if formato_legado(arq):
    warnings.warn("formato legado")
    # programa CONTINUA
    converter(arq)
AFTER
if formato_legado(arq):
    raise ValueError("formato legado")
    # programa PARA aqui

warn→raise transforma um aviso em erro fatal. Variações do mesmo problema: trocar o tipo da exceção (RuntimeError→ValueError muda quem a captura), mudar a mensagem (quebra teste e parsing de log), apertar ou afrouxar um except.

grep mental por raise, warn, except nos dois lados: mesmos tipos, mesmas condições, mesmas mensagens?

A6 — Falsas mudanças: equivalências que parecem quebra NÃO rejeitar

O item que protege na direção contrária — na rodada 1 matamos pares válidos por estes três padrões:

PARECE mudança…
if x < 0:
    return "neg"
elif x == 0:      # virou if
    return "zero"
…mas é equivalente
if x < 0:
    return "neg"
if x == 0:
    return "zero"
# só se chega à 2ª checagem se a 1ª
# NÃO retornou — o elif era redundante
PARECE mudança…
def itens_validos(seq):
    for s in seq:
        if s is None:
            return   # era "pass" + break
        yield s
…mas é equivalente
# dentro de um GENERATOR, "return" vazio
# não devolve valor a ninguém: só encerra
# a iteração (StopIteration), exatamente
# como sair do loop e cair no fim
PARECE mudança…
if not valido(x):
    raise ValueError(x)
else:             # else removido
    processar(x)
…mas é equivalente
if not valido(x):
    raise ValueError(x)
processar(x)
# depois de um raise/return, o else
# é decorativo — dedentar é seguro
antes de cravar muda_comportamento em mudança de fluxo de controle, pergunte: o ramo anterior termina (return, raise, continue)? Se sim, elif→if, remoção de else e dedent são equivalentes. E return vazio em generator não é "retornar None para o chamador" — é só parar de iterar.

Eixo B — o after é uma boa aula? (eliminatório)

Pergunta-guia: "se um modelo imitar exatamente este diff, ele aprende a resolver este smell — e só isso?" Três sub-checagens, nesta ordem: (1) o before tem o smell rotulado de verdade? (2) a resolução passou do limiar (não é só mitigação)? (3) o diff ficou no escopo do smell?

R1 — Long Method → Extract Method

RESOLVE genuína
def importar(arq):
    dados = _ler_e_validar(arq)
    regs  = _converter(dados)
    _gravar(regs)
# principal virou orquestração:
# passos nomeados, 1 nível de abstração
SÓ MITIGA lixo / after_nao_resolve
def importar(arq):
    dados = _ler_e_validar(arq)   # extraiu 1 bloco…
    # --- converte registros ---
    regs = []
    for d in dados: ...           # 20 linhas
    # --- grava no banco ---
    con = abrir(); ...            # 15 linhas
# …e o resto continua demarcado
# por comentários-seção
Não puna o lado oposto: (a) extração que elimina um bloco duplicado 2×+ é resolução material mesmo se a função continua comprida — eliminar duplicação é a motivação canônica de Extract Method; (b) não exija quebrar uma sequência coesa de algoritmo em pedacinhos: over-extraction também é defeito. Sinal do before legítimo: responsabilidades distintas identificáveis (seções comentadas, duplicação, fases preparar/processar/finalizar) — comprimento por boilerplate/aridade NÃO é Long Method.

R2 — Long Parameter List → Parameter Object

RESOLVE genuína
@dataclass
class ExportConfig:
    formato: str
    compressao: str = "gzip"
    incluir_meta: bool = True

def exportar(dados, cfg: ExportConfig):
    arq = abrir(cfg.formato)      # acesso no
    ...                           # ponto de uso
DISFARÇA lixo
def exportar(dados, **kwargs):
    formato    = kwargs["formato"]
    compressao = kwargs["compressao"]
    incluir    = kwargs["incluir_meta"]
    ...
# o smell não sumiu: foi escondido num
# dict sem tipo. E o desempacotamento em
# massa no topo denuncia bundle-sacola
Limiar: assinatura final ≤5 parâmetros com objeto coeso nomeado pelo domínio (13→10 é mitigação). A definição do objeto deve aparecer no after (senão é A2). O sujeito/payload da função (ex.: dados) fica fora do objeto — agrupa-se configuração com configuração, contexto com contexto.

R3 — Magic Numbers → Named Constant

RESOLVE genuína
MIN_ANO_VALIDO = 1900
MAX_ANO_VALIDO = 2030

def validar(ano):
    if not (MIN_ANO_VALIDO <= ano
            <= MAX_ANO_VALIDO):
        raise ValueError(ano)
# definição visível + as DUAS pontas
# do mesmo conceito nomeadas juntas
PELA METADE lixo / after_nao_resolve
PIVO_SECULO = 90

def expandir(aa):
    if aa > PIVO_SECULO:
        return 1900 + aa
    return 2000 + aa
# nomeou o pivô e deixou as bases 1900 e
# 2000 — metade do MESMO conceito.
# Resolução incoerente
O que NÃO é resíduo (não conte contra o after): literais estruturais — índice, passo, acumulador (0, 0.0, i + 1, range(len(xs) - 1)) — e idiomáticos autoexplicativos (date(ano, mes, 1) = primeiro dia). Nomeá-los seria ruído. O diff válido de R3 é exclusivamente: +definições e trocas 1-para-1 literal→nome. Mudou assinatura, fluxo ou algoritmo junto? Escopo estourado.

R4 — Deep Nesting → Guard Clauses

RESOLVE genuína
def enviar(msg, canal):
    if canal is None or not canal.aberto:
        return
    if not msg:
        return
    for parte in msg.partes:
        if not parte.valida:
            continue
        canal.push(parte)
# profundidade máx: 5 → 2; happy path
# no nível mais raso
COSMÉTICO lixo / after_nao_resolve
def enviar(msg, canal):
    if canal is not None:
        if canal.aberto:
            if msg:
                for parte in msg.partes:
                    if parte.valida:
                        canal.push(parte)
    # antes havia um "else: return" aqui,
    # que foi removido
# a pirâmide continua intacta:
# profundidade máxima inalterada
Limiar objetivo: a profundidade máxima de indentação do caminho principal cai ≥1 nível. Conte! Técnica-alvo: inversão de condição + return/raise/continue cedo. Se o aninhamento sumiu porque um bloco virou função, a técnica é Extract Method (R1), não R4. E elif-ladder de despacho (if tipo == "a": … elif tipo == "b": …) não é deep nesting — é outro smell, não conte como before válido.

R5 — Dead Code → Remove

RESOLVE genuína
# BEFORE
fig = plotar(dados)   # fig nunca é lida

# AFTER
plotar(dados)
# removeu SÓ o binding morto e manteve a
# chamada — plotar() tem efeito colateral
# (desenha); deletar a linha inteira
# mudaria o comportamento
BUG DISFARÇADO lixo / outro
# BEFORE
def total(itens):
    soma = sum(i.preco for i in itens)
    # soma nunca usada; função devolve None

# AFTER
def total(itens):
    soma = sum(i.preco for i in itens)
    return soma
# o "dead store" era SINTOMA de um return
# esquecido. O commit consertou o bug —
# muda o comportamento (None → número).
# Não é Dead Code → Remove
Teste de intenção, sempre primeiro: o valor morto obviamente deveria ser retornado/propagado? Então é bug, não smell (foi 30% da nossa amostra R5). Limiar de resolução: diff = exclusivamente remoções, cobrindo o fecho transitivo — se remover x torna y morto também, y sai junto; deixar resíduo da mesma classe é resolução parcial.

Colinha final — fluxo por par (~5 min)

PassoO quêSe falhar
1Ler na ordem: def → returns → nomes novos → corpo—
2Eixo A em voz alta: A1 assinatura · A2 símbolos · A3 curto-circuito · A4 early-return×tail · A5 exceções (A6: não rejeitar equivalências)lixo / muda_comportamento — fim, não debater B
3B1: before tem o smell rotulado?lixo / before_sem_smell → descartar

incerto é reservado para inverificabilidade (símbolo/efeito fora do recorte — A2). "Resolveu em parte" não é incerto, é lixo com before aproveitável. Persistiu o desacordo entre nós dois? → lixo conservador, com o dissenso registrado.
